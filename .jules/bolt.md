## 2024-05-18 - [Parallelize Bulk Message Sending Endpoint]
**Learning:** Refactoring a sequential `for...of` loop to use `Promise.all` for bulk message processing must be combined with chunking (concurrency limits). Direct mapping of user input (e.g., `req.body`) to parallel tasks creates unbounded concurrency, leading to resource exhaustion, `EMFILE` errors, or triggering rate-limit bans when hitting external APIs or processing many queues simultaneously.
**Action:** When parallelizing operations on user-provided arrays in Node.js, always implement a `CONCURRENCY_LIMIT` and process the array in smaller chunks to protect server resources and maintain system stability.
## 2024-06-09 - [Dashboard Analytics Optimization]
**Learning:** The dashboard was missing a core `getAnalytics` method in the `ActivityLog` model, and the `/analytics` route was calling it without `await`, leading to unhandled promises or empty responses. Implementing time-series grouping directly in SQL (`GROUP BY date(created_at)`) significantly reduces the overhead compared to fetching all logs and grouping in JavaScript, especially as the `activity_logs` table grows.
**Action:** Always prefer SQL-level aggregation for dashboard charts. Combined with a composite index on `(action, created_at, user_email)`, this ensures O(1) query time relative to the number of logs for a fixed time window.
